## 1. FASTQ 文件

**是什么**：测序仪的原始输出。DNA 被打碎成 ~150bp 的小片段，测序仪逐个读出碱基序列。每条记录包含：
- 片段 ID
- 碱基序列（ATCG...）
- 每个碱基的质量分数（测序仪对自己读得准不准的信心）

**为什么一个人有两个文件（R1 + R2）**：
双端测序（paired-end），从 DNA 片段的两头各读一次。R1 和 R2 一一对应，比对时能更准确定位到基因组位置。

**对应代码**：`app/pipeline/alignment.py` → `AlignmentModule.align(fastq_r1, fastq_r2, ...)`

**古 DNA 特殊性**：
- 样本降解严重，片段更短，覆盖度更低
- 系统中 yleaf 默认 `reads_thresh=1`，就是为了适应古 DNA 低覆盖度

## 2. 比对（Alignment）

**是什么**：把 FASTQ 里几亿条碎片逐条"贴回"参考基因组，找到每条片段来自哪个位置，输出 BAM 文件。

**耗时**：最吃算力的步骤。30x 全基因组 8 线程要 6-10 小时，古 DNA 覆盖度低会快很多。

**对应代码**：`app/pipeline/alignment.py` → BWA-MEM + samtools sort

**参考基因组只有 3 个主力版本**：

| 版本 | 别名 | 年份 | 说明 |
|---|---|---|---|
| hg19 | GRCh37 | 2009 | 老主力，大量历史数据 |
| hg38 | GRCh38 | 2013 | 当前主流 |
| T2T-CHM13v2 | hs1 | 2022 | 第一个完整基因组 |

**关键理解**：
- 比对时选哪个参考基因组，BAM 就是哪个坐标系
- 不同版本之间靠 liftOver（chain 文件）做坐标转换，不用重新比对
- liftOver 95%+ 位点能精确转换，少量复杂区域会丢失，正常现象

**产品需求**：
- 用户提交 FASTQ 时可以选择比对到 hg19、hg38 或 T2T，系统需要三套参考基因组都准备好
- 用户上传已比对好的 BAM 时，系统自动读取 header 识别参考基因组版本（hg38/hg19/GRCh37/T2T），不需要用户手动选择。识别方法：匹配 chr1 长度 + 染色体命名风格，100% 可靠

## 3. BAM 文件与参考基因组识别

**BAM 是什么**：比对后的结果文件，记录了每条 DNA 碎片落在参考基因组的什么位置。

**不同样本 BAM 大小差异很大**：
- 30x 覆盖度的现代人 BAM 可能几十 GB
- 0.1x 覆盖度的古人 BAM 可能只有几百 MB
- 大小取决于测序深度，跟参考基因组版本无关

**如何识别 BAM 的参考基因组版本**：
- BAM 文件头（header）里记录了比对时用的参考基因组信息
- header 在文件开头，读取瞬间完成，不用扫描整个文件
- 关键字段：`@SQ SN:chr1 LN:248956422` — 染色体名称和长度
- 这个长度是参考基因组的长度，不是样本的，跟覆盖度无关
- 不同版本的 chr1 长度是唯一确定的：

| chr1 长度 | 版本 |
|---|---|
| 248,956,422 | hg38 (GRCh38) |
| 249,250,621 + chr 前缀 | hg19 |
| 249,250,621 + 无 chr 前缀 | GRCh37/hs37d5 |
| 248,387,328 | T2T-CHM13v2 |

**比方**：参考基因组是一面完整的墙，DNA 碎片是往墙上扔泥巴。不管扔了多少泥巴，墙的尺寸不变。BAM header 记录的是墙的规格。

## 4. BAM 是系统的分水岭

**BAM 之前**：体力活，只干一件事——FASTQ 比对成 BAM。耗时长，逻辑简单。

**BAM 之后**：分析活，分支多，每个分支有自己的工具和参考数据：
```
BAM（全基因组）
  ├─→ chrY   → Y 单倍群（yleaf）
  ├─→ chrM   → MT 单倍群（Haplogrep）
  ├─→ 常染色体 → 芯片格式导出 / 1240K 提取 → PLINK 群体分析
  └─→ chrX   → (暂不涉及)
```

**BAM 是核心资产**：
- 比对花几小时，结果必须留着
- 后续分析可以反复跑、换参数、加新类型，都从 BAM 开始
- 用户今天跑 Y 单倍群，过两天想跑 1240K，不用重新比对

**不同参考系的 BAM 提取出来的数据有差异**：
- 坐标不同（同一个 SNP 门牌号不一样）
- 能比上的读取数略有不同
- 但最终生物学结论一致（单倍群判定结果一样）

### 核心设计决策：三套参考系各自独立，不做坐标转换

**需求**：不依赖 liftOver 坐标转换，而是给 hg38、hg19、T2T 各准备一套完整的独立流程。

**理由**：
- 坐标转换有少量丢失，不如原生坐标可靠
- 每个版本的 SNP 位点文件、注释文件本来就是独立的
- 流程逻辑完全一样，只是参考数据不同
- 用户选了哪个参考系，就走那套，干净利落

**架构**：
```
用户选择参考系（或 BAM 自动识别）
  │
  ├─→ hg38 流程：hg38/genome + hg38/snp + hg38/y_chromosome + hg38/mt + hg38/1240k
  ├─→ hg19 流程：hg19/genome + hg19/snp + hg19/y_chromosome + hg19/mt + hg19/1240k
  └─→ t2t 流程： t2t/genome  + t2t/snp  + t2t/y_chromosome  + t2t/mt  + t2t/1240k
```

三套流程代码一样，只是传入的 reference 版本不同，系统自动去对应文件夹找数据。

> liftOver 坐标转换只在特殊场景保留（比如芯片格式导出需要 hg19 坐标的模板），不作为主流程依赖。
